# 북미 서비스 라우팅 프로젝트 설계서

## 1. 목적

북미 지역의 서비스 데이터를 기반으로 최적 경로 라우팅 알고리즘을 개발한다.

프로젝트는 다음 순서로 진행한다.

1. 알고리즘 설계
2. 시뮬레이션 수행
3. 성능 검증
4. 실제 서비스 형태의 지도 기반 배치 및 시각화

## 2. 기본 원칙

- 현재 데이터는 최종본이 아니므로 언제든 교체될 수 있다.
- 따라서 초기 단계에서는 데이터 내용보다 폴더 구조와 처리 파이프라인 구조를 우선 확정한다.
- 전체 구조는 `D:\python\SM 라우팅` 프로젝트의 한국서비스 라우팅 구조를 기준으로 가져간다.
- 핵심 비즈니스 로직은 현재 프로젝트의 `smart_routing` 폴더에 배치한다.

## 3. 디렉터리 구조 원칙

프로젝트 루트 기준 목표 구조는 아래와 같다.

```text
북미 라우팅/
  data/
  260310/
    input/
    output/
  smart_routing/
  templates/

```

## 4. 데이터 저장 규칙

### 4.1 고정 데이터

`data` 폴더에는 변동이 적은 기준 데이터를 저장한다.

예시:

- 우편번호 구역 데이터
- SM 할당 지역 데이터
- SM 수리 제품 데이터
- 기타 마스터성 기준 데이터

### 4.2 날짜별 가변 데이터

날짜 폴더 예: `260310`

해당 폴더에는 일자별로 달라질 수 있는 원천 데이터를 저장한다.

예시:

- 서비스 실적 데이터
- SM 타임스케줄 데이터

### 4.3 전처리 결과

전처리 완료 후 산출물은 날짜 폴더 하위 `input` 폴더에 저장한다.

예시:

- 정규화 서비스 데이터
- 정규화 타임테이블 데이터
- 정규화 운영지역 데이터
- 시뮬레이션 입력용 SM 위치/권역 데이터

### 4.4 시뮬레이션 및 분석 결과

시뮬레이션 결과 및 성능 검증 결과는 날짜 폴더 하위 `output` 폴더에 저장한다.

예시:

- 배정 결과
- 타임테이블 결과
- 거리 및 시간 비교 결과
- 통계 리포트
- 지도 서비스용 결과 데이터

## 5. 처리 파이프라인

한국서비스 프로젝트 기준으로 북미 프로젝트도 아래 4단계 파이프라인을 따른다.

### 5.1 전처리

원천 데이터를 정규화하고 시뮬레이션 입력 구조로 변환한다.

주요 처리 대상:

- 서비스 데이터 정리
- SM 스케줄 정리
- 우편번호/권역 매핑
- 제품/기술 역량 매핑
- 주소 기반 위도/경도 생성

#### 주소 지오코딩 정책

- 미국 주소의 위도/경도 생성은 `US Census Geocoder`를 기본으로 사용한다.
- 1차 변환은 `US Census Geocoder`, 2차 fallback은 `Google Geocoding API`를 사용한다.
- 서비스 원천 데이터에는 위도/경도 컬럼이 없을 수 있으므로 전처리 전에 주소 지오코딩 단계를 수행한다.
- 지오코딩 결과는 매번 다시 호출하지 않고 캐시 파일에 누적 저장한다.
- `US Census Geocoder`의 배치 주소 변환 한도에 맞춰 하루 최대 10,000건만 신규 변환한다.
- 실제 호출은 타임아웃을 피하기 위해 더 작은 배치 단위로 나누어 전송한다.
- 이미 변환한 주소는 캐시를 재사용하고, 미변환 주소만 다음 실행일에 이어서 추가 변환한다.
- Google API는 Census 미해결 주소에만 제한적으로 사용해 비용을 최소화한다.
- Google API로 한 번이라도 시도한 주소는 별도 시도 이력 파일에 기록하고 재실행 시 다시 호출하지 않는다.
- Google API는 월간 시도 건수를 별도 집계하고, 설정된 월 상한을 초과하지 않도록 실행 단계에서 차단한다.
- 지오코딩 전 주소 전처리를 수행해 `ADDRESS_LINE1_INFO`에 중복 포함된 도시, 주, 우편번호, 국가 정보를 제거한 뒤 요청한다.
- 주소 정제 규칙이 변경된 경우에 한해 운영자가 1회성 재시도 옵션으로 기존 Google 시도 이력을 무시할 수 있다.
- 이 1회성 재시도 이후의 결과도 다시 시도 이력에 기록되므로, 이후 실행에서는 다시 재호출하지 않는다.
- 최종 병합 결과에서 위도 또는 경도가 비어 있는 행은 `source=failed`로 표기해 실패 행을 즉시 구분할 수 있게 한다.
- 실행 로그는 전처리/지오코딩 단계 수치와 최종 병합 결과 수치를 구분해 출력한다.
- 지오코딩 결과에는 최소한 다음 정보를 저장한다.
  - 원본 주소 키
  - 매칭 주소
  - 위도
  - 경도
  - 매칭 상태
  - Census 지역 코드

#### 북미 서비스 데이터 주의사항

- 서비스 파일은 접수번호 기준 1행이 아닐 수 있으며 부품 기준으로 여러 행이 반복될 수 있다.
- 따라서 시뮬레이션 입력용 서비스 데이터 생성 전 `GSFS_RECEIPT_NO` 기준 집계 규칙이 필요하다.
- 주소 컬럼은 현재 다음 영문 컬럼을 기준으로 사용한다.
  - `ADDRESS_LINE1_INFO`
  - `CITY_NAME`
  - `STATE_NAME`
  - `POSTAL_CODE`
  - `COUNTRY_NAME`
- 미국 이외 국가 데이터가 들어오면 `US Census Geocoder` 대상에서 제외하고 별도 정책으로 처리한다.

### 5.2 공간/권역 모델 생성

라우팅에 필요한 권역, 활동범위, SM 위치 기준 데이터를 생성한다.

### 5.3 시뮬레이션

전처리 산출물을 기반으로 라우팅 배정 시뮬레이션을 수행한다.

### 5.4 분석 및 서비스 시각화

시뮬레이션 결과를 성능 지표로 검증하고, 실제 서비스처럼 지도 기반 화면에 배치한다.

### 지역 검토 지도

- 북미 지역 검토용 지도는 미국 Census ZCTA(우편번호 경계) 지리정보를 사용한다.
- 한국 프로젝트처럼 사전 생성된 운영지역 geometry가 없으므로 우편번호 경계를 기준으로 area geometry를 도출한다.
- 지도 앱은 미국 전체 범위를 기본 대상으로 하며, `전체` 또는 개별 `STRATEGIC_CITY_NAME` 필터를 제공한다.
- 화면은 좌우 분할 레이아웃으로 구성한다.
  - 좌측: 도시/지역 선택 콤보박스, 통계표
  - 우측: 지도
- 지도는 최소 두 레이어를 제공한다.
  - ZIP Coverage
  - AREA Coverage
- 추가로 geocoded 서비스 좌표를 점 레이어로 표시한다.
- 지역 팝업/요약에는 최소 다음 정보를 포함한다.
  - `AREA_NAME`
  - `SVC_ENGINEER_CODE`
  - `SVC_CENTER_TYPE`
- `AREA_NAME`별 색상을 다르게 적용해 지역 경계를 시각적으로 구분한다.
- 선택한 도시 범위에서 지역에 포함되지 않은 ZIP 경계는 흐린 회색 배경 레이어로 표시한다.
- 좌측 통계표에는 최소 다음 값을 포함한다.
  - 지역별 우편번호 수
  - 지역별 서비스 수
  - 지역별 면적 합계(km²)
- 면적이 0으로 보이는 경우는 실제 면적이 0이 아니라 해당 ZIP이 Census ZCTA geometry와 매칭되지 않은 경우일 수 있다.
- `AREA_NAME`이 여러 ZIP을 포함하므로 area geometry는 ZIP geometry union/dissolve 방식으로 만든다.
- 한 ZIP이 여러 `AREA_NAME`에 중복 배정될 수 있으므로, 지도는 현재 배정 상태 검토용 시각화로 해석한다.

## 7. 문서 관리 규칙

- 설계 내용은 `docs` 폴더에 문서로 남긴다.
- 작업 구조, 정책, 설계 변경 시 설계서를 계속 수정한다.
- Codex가 변경한 내용은 매 작업마다 루트의 `UPDATED_BY_CODEX.md`에 기록한다.

## 8. 현재 상태

- 현재는 최종 데이터 설계 전 단계이다.
- 지금은 데이터 정합성보다 전체 구조와 처리 흐름을 먼저 확정하는 단계이다.
- 이후 실제 최신 데이터를 다시 저장한 뒤 전처리 로직과 컬럼 매핑을 북미 기준으로 구체화한다.
