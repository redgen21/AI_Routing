# 북미 라우팅 프로젝트 설계서

## 1. 목적

북미 서비스 데이터를 기준으로 기사 배정과 방문 순서를 계산하고, 이를 검토용 지도, 시뮬레이션 결과, 운영 API 형태로 재사용할 수 있는 라우팅 플랫폼을 구축한다.

현재 프로젝트는 다음 2개 축으로 운영된다.

1. 애틀랜타 중심의 생산/시뮬레이션 라우팅
2. 공통모드(Common VRP Mode) 기반의 범용 클라이언트/서버 라우팅

공통모드는 특정 도시 전용 스크립트에 묶이지 않고, 도시/법인/기사/서비스 입력을 공통 구조로 정규화한 뒤 같은 API와 같은 저장 구조로 처리하는 것을 목표로 한다.

## 4. 공통 처리 파이프라인

프로젝트의 공통 흐름은 아래 순서를 따른다.

1. 원본 서비스/기사 입력 수집
2. 주소 및 기사 시작점 지오코딩
3. 공통 입력 스키마로 정규화
4. 라우팅 payload 생성
5. 비동기 라우팅 실행
6. 결과 저장
7. 지도/비교/운영 화면에서 재사용

## 5. 공통모드 아키텍처

### 5.1 개요

공통모드는 다음 네 레이어로 구성된다.

1. 클라이언트 레이어
2. 공통 API 서버 레이어
3. 공통 DB 및 런타임 레이어
4. 실제 라우팅 엔진 레이어

### 5.2 클라이언트 프로그램

공통모드 클라이언트의 진입점은 아래 파일이다.

- `sr_common_vrp_client.py`

이 클라이언트는 Streamlit UI를 제공하며 다음 역할을 담당한다.

- 법인, 도시, 일자 선택
- 서비스 입력 로드 및 저장
- technician availability 편집
- 공통 API 서버 호출
- payload 디버그 정보 확인
- 최신 라우팅 결과 조회
- 지도에서 지역/기사/경로 시각화

또한 라우팅 결과 시각화를 위해 다음 보조 로직을 사용한다.

- `smart_routing.area_map.load_city_map_data()`
- `smart_routing.osrm_routing.OSRMTripClient`
- 지역 ZIP/폴리곤 레이어 생성
- 기사 홈 좌표와 방문 경로 오버레이

### 5.3 서버 프로그램

공통모드 서버의 진입점은 아래 파일이다.

- `sr_common_vrp_api_server.py`
- `restart_common_vrp_api.sh`

실제 HTTP 서버 구현은 아래 모듈에 있다.

- `smart_routing/common_vrp_api_server.py`

서버는 Python 표준 라이브러리 `ThreadingHTTPServer` 기반으로 동작하며 기본 포트는 `8065`다.

`config_common_vrp.json` 기준 현재 기본 설정은 다음과 같다.

- host: `0.0.0.0`
- port: `8065`
- routing_api_url: `http://20.51.244.68:8065`

### 5.4 공통 DB/런타임 레이어

공통모드의 핵심 저장 및 실행 레이어는 다음 모듈이다.

- `smart_routing/common_vrp_db.py`
- `smart_routing/common_vrp_runtime.py`

역할은 다음과 같다.

- 공통 설정 로드
- PostgreSQL 연결
- 공통 마스터/입력/요청/결과 테이블 관리
- 기사 home geocode 보강
- 서비스/기사 입력을 API payload로 변환
- 중수리 여부와 서비스 시간 보강
- 요청/결과를 DB와 파일 시스템에 동시 저장

### 5.5 실제 라우팅 엔진 레이어

공통모드가 최종적으로 호출하는 비동기 라우팅 엔진은 아래 모듈이다.

- `smart_routing/vrp_api_service.py`

이 모듈은 요청 payload를 job 폴더에 저장하고, 모드별 handler를 선택해 실행한다.

현재 모드 매핑은 다음과 같다.

- `na_general` -> `smart_routing.vrp_mode_na_general`
- `z_weekend` -> `smart_routing.vrp_mode_z_weekend`

`vrp_api_common.py`에는 alias와 지원 모드 정규화 규칙이 정의되어 있으며, 현재 시스템적으로는 아래 값들을 지원 대상으로 본다.

- `na_general`
- `weekday_general`
- `z_weekday`
- `z_weekend`

다만 실제 handler가 구현되어 있는 것은 현재 `na_general`, `z_weekend` 중심이다.

## 6. 공통모드 서버 API 설계

### 6.1 POST 엔드포인트

공통 API 서버는 현재 아래 POST 엔드포인트를 제공한다.

- `/api/v1/common/init`
  - 기본 마스터 seed 생성
- `/api/v1/common/jobs/bulk_upsert`
  - 서비스 입력 bulk upsert
- `/api/v1/common/technicians/replace`
  - 특정 일자의 technician availability 전체 교체
- `/api/v1/common/routing-config/upsert`
  - 도시별 라우팅 설정 upsert
- `/api/v1/common/routing/build-payload`
  - 서비스/기사 입력을 공통 routing payload로 빌드
- `/api/v1/common/routing/run`
  - payload 생성 후 비동기 라우팅 job 시작
- `/api/v1/common/routing/check`
  - request_id 기준 현재 상태와 결과 재조회

### 6.2 GET 엔드포인트

현재 GET 엔드포인트는 다음과 같다.

- `/api/v1/common/contexts`
- `/api/v1/common/engineers`
- `/api/v1/common/capabilities`
- `/api/v1/common/jobs`
- `/api/v1/common/technicians`
- `/api/v1/common/regions`
- `/api/v1/common/routing-config`
- `/api/v1/common/routing/latest`

이 엔드포인트들은 공통 UI와 운영 스크립트가 같은 방식으로 데이터를 조회할 수 있도록 설계되어 있다.

## 8. 공통 payload 생성 로직

`smart_routing/common_vrp_runtime.py`의 payload 생성 로직은 아래 순서로 동작한다.

1. 서비스 행과 technician 행을 DataFrame으로 변환
2. available technician만 선별
3. 기사 마스터와 라우팅 설정 조회
4. 기사 시작점이 `Home`이면 home 좌표 사용
5. 기사 시작점이 `Custom Address`이면 지오코딩 수행
6. 기사 시작 좌표 기준으로 서쪽에서 동쪽 순 정렬
7. technician payload 생성
8. job payload 생성
9. request_id, mode, planning_date, options 포함 최종 payload 반환

현재 payload 기본 옵션은 아래와 같다.

- `respect_fixed_jobs = true`
- `objective = min_total_travel_time`
- `time_limit_seconds = 30`
- `timezone_offset = routing config 값`

job payload에 포함되는 핵심 정보는 다음과 같다.

- `salesforce_id`
- `receipt_no`
- `product_group`
- `product`
- `symptom`
- `address`
- `city_name`
- `state_name`
- `country_name`
- `postal_code`
- `location`
- `fixed`
- `current_employee_code`
- `current_center_type`

technician payload에 포함되는 핵심 정보는 다음과 같다.

- `employee_code`
- `employee_name`
- `center_type`
- `start_location`
- `end_location`
- `shift_start`
- `shift_end`
- `slot_count`
- `max_jobs`

## 9. 중수리 및 서비스 시간 로직

공통모드에서는 job 생성 후 중수리 여부를 추가 판단한다.

판단 기준은 아래 데이터다.

- DB의 `common_heavy_repair_rule_master`
- fallback 파일 `atlanta_heavy_repair_lookup.csv`
- 추가 fallback 생성 로직

판단 결과는 다음과 같이 payload에 반영된다.

- `is_heavy_repair = true/false`
- `service_minutes = 100` if heavy repair
- `service_minutes = 45` otherwise

즉, 공통모드도 애틀랜타 생산 라우팅에서 쓰던 45/100분 규칙을 재사용하도록 설계되어 있다.

## 10. 비동기 job 처리 구조

비동기 라우팅 실행은 `smart_routing/vrp_api_service.py`가 담당한다.

### 10.1 저장 위치

각 job은 아래 폴더에 개별 디렉터리로 저장된다.

- `260310/vrp_api_jobs/<job_id>/`

저장 파일은 다음과 같다.

- `request.json`
- `status.json`
- `result.json`
- `error.txt`

### 10.2 상태 전이

job 상태는 대략 아래 순서로 바뀐다.

1. `queued`
2. `running`
3. `completed` 또는 `failed`

### 10.3 처리 흐름

1. `submit_routing_from_payload()`가 job id 생성
2. request payload를 `request.json`에 저장
3. 상태를 `queued`로 저장
4. background thread에서 `process_job()` 실행
5. mode별 handler를 선택해 라우팅 수행
6. 결과를 `result.json`과 DB result 테이블에 저장
7. 상태를 `completed` 또는 `failed`로 갱신

## 11. 라우팅 모드별 역할

### 11.1 `na_general`

북미 일반 배정용 공통 모드다.

역할은 다음과 같다.

- 공통 technician/job payload를 실제 내부 서비스 프레임으로 변환
- 지역 ZIP 기준 region assignment 보강
- 기사 시작점과 서비스 포인트를 함께 사용해 경로 계산
- fixed job을 유지한 상태로 배정 수행
- OR-Tools 기반 경로 계산 결과를 공통 API 응답 형식으로 변환

### 11.2 `z_weekend`

한국 weekend Z-job 스타일에 가까운 배정 규칙을 공통 API 체계에서 재사용하기 위한 모드다.

역할은 다음과 같다.

- 고정된 시간 슬롯 기반 배정
- 슬롯 순번/시간 텍스트 계산
- 필요 시 OSRM route distance fallback 사용

## 12. 공통 클라이언트 동작 흐름

`sr_common_vrp_client.py` 기준 UI 동작은 아래 순서를 따른다.

1. 공통 설정과 서버 URL 로드
2. 법인/도시/일자 선택
3. 입력 서비스 파일 또는 저장된 parquet 로드
4. technician availability 구성
5. 서버의 build-payload 호출
6. 디버그 정보 확인
7. 서버의 run 호출
8. request_id 기준 상태 polling 또는 latest snapshot 조회
9. 결과를 표와 지도에서 확인

지도 화면에서는 다음 정보를 중점적으로 보여준다.

- region polygon
- ZIP coverage
- 기사 home start 위치
- 기사별 route polyline
- 방문 순번 마커
- 지역별 staffing 현황

## 14. 공통모드와 기존 전용 모드의 관계

공통모드는 기존 애틀랜타 전용 라우팅을 대체하기보다, 아래 목적의 공통 운영 레이어다.

1. 입력 구조를 표준화
2. API 호출 방식을 통일
3. DB 저장과 재조회 구조를 통일
4. 도시/법인 확장을 쉽게 함

즉, `production_assign_atlanta*` 계열은 도시 전용 상세 로직을 유지하고, 공통모드는 이를 감싸는 범용 인터페이스와 공통 저장소를 제공하는 구조다.

## 15. 현재 설계 기준 주요 제약

- 공통모드 alias는 여러 개 정의되어 있지만 실제 mode handler는 일부만 구현됨
- 기사 시작점 geocoding 실패 시 payload 생성 단계에서 요청이 중단됨
- 공통모드도 여전히 애틀랜타 기준 참조 데이터와 규칙을 일부 공유함
- 지역 기준과 capability 규칙은 도시가 늘어나면 별도 seed 전략이 더 필요함
- `260310/vrp_api_jobs` 파일 저장 방식은 단순하고 강하지만, 장기적으로는 보관/정리 정책이 필요함

- capability 이력 관리

## 17. 문서 관리 규칙

- 구조와 설계 변경은 `docs/` 아래 문서에 반영
- Codex가 변경한 작업 이력은 루트 `UPDATED_BY_CODEX.md`에 기록
- 공통모드 구조가 바뀌면 본 설계서를 우선 업데이트

## 18. 현재 상태 요약

현재 프로젝트는 단순 스크립트 묶음을 넘어, 다음 요소를 가진 공통 라우팅 플랫폼으로 발전한 상태다.

- Streamlit 기반 공통 클라이언트
- 표준 HTTP 공통 서버
- PostgreSQL 기반 마스터/입력/결과 저장소
- 파일 기반 비동기 job 처리
- 모드별 라우팅 엔진 분기
- 지도 기반 결과 검토 UI

즉, 공통모드는 북미 라우팅 프로젝트에서 클라이언트, 서버, 저장소, 라우팅 로직을 하나의 일관된 운영 체계로 묶는 중심 계층이다.
